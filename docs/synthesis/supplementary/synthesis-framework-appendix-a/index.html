Appendix A: Archetype Library

How to Use This Appendix

Archetypes are pre-validated scaffold patterns—structural templates that have been proven to work across multiple games. Think of them as starting points that encode accumulated design knowledge.

When to use archetypes:

  • You're new to game design and want a proven pattern
  • You want to save time by starting from a validated structure
  • You're working in a familiar genre and want best practices
  • You want to understand what makes a pattern work before modifying it

When NOT to use archetypes:

  • You're innovating and need a truly custom structure
  • Your aesthetic goals don't match available archetypes
  • You want to learn by building from first principles
  • Your constraints make existing patterns unworkable

How to use an archetype:

  1. Find the archetype closest to your goals
  2. Read the complete template
  3. Note what you'll keep vs. change
  4. Proceed through the framework using the archetype as reference
  5. Justify any changes based on your aesthetic goals

Modifying archetypes:

  • Keep what serves your aesthetics
  • Change what doesn't
  • Document why you're diverging (helps catch mistakes)
  • Test modified patterns thoroughly

Archetype Template Structure

Each archetype in this appendix follows this structure for consistency:

Overview: Brief description and core appeal

Primary Aesthetics: What experiences this pattern creates

Player Profile Match: Who this works for (and who it doesn't)

Temporal Structure: Loop architecture or narrative pattern

Common Mechanics: Typical verbs and interaction models

Resource Patterns: Standard resource types and flows

Content Needs: How much content and what type

Platform Considerations: Where this pattern works best

Monetization Patterns: Common business models (if applicable)

Social Features: Typical multiplayer/social elements (if any)

Success Metrics: KPIs to track

Common Variations: How this pattern is often modified

Design Pitfalls: What typically goes wrong

Example Games: Real implementations of this archetype

When This Archetype Fails: Conditions where pattern doesn't work

Using the Archetype Library

When to Start from an Archetype

Start from an archetype when:

  • You're new to game design (proven patterns reduce risk)
  • You're working in a familiar genre (roguelike, daily puzzle, etc.)
  • You want to save time (templates accelerate decisions)
  • You're unsure how much content you need (archetypes provide estimates)
  • You want to avoid common pitfalls (each template lists them)

When to Build Custom

Build custom when:

  • Your aesthetic goals don't match any archetype
  • You're deliberately innovating (breaking genre conventions)
  • You're combining multiple archetypes (hybrid design)
  • Your constraints make existing patterns unworkable
  • You want to learn by building from first principles

How to Modify an Archetype

Modification Process:

  1. Choose the archetype closest to your vision
  2. Identify what serves your aesthetics (keep this)
  3. Identify what doesn't (change or remove this)
  4. Document your changes and why
  5. Run through validation (Phase 5) carefully
  6. Test modified pattern thoroughly (more risk than pure archetype)

Common Modifications:

  • Change temporal structure (compressed loops instead of full stack)
  • Adjust difficulty (casual roguelike instead of hardcore)
  • Add/remove mechanics (deckbuilding + metroidvania exploration)
  • Change monetization (premium instead of F2P)
  • Shift primary aesthetic (horror walking sim instead of contemplative)

Cross-Referencing with Framework Phases

Each archetype component maps to framework phases:

Archetype Section → Framework Phase:

  • Primary Aesthetics → Phase 1 (Aesthetic Crystallization)
  • Player Profile → Phase 1 (Aesthetic Crystallization)
  • Temporal Structure → Phase 2 (Scaffold Architecture)
  • Common Mechanics → Phase 3 (Mechanical Specification)
  • Resource Patterns → Phase 3 (Mechanical Specification)
  • Content Needs → Phase 4 (Content Specification)
  • Design Pitfalls → Phase 5 (Validation)

Use archetypes as reference while working through framework phases.

Missing Your Archetype?

These 10 archetypes cover common patterns, but aren't exhaustive. If your game doesn't fit any:

Similar Archetypes Not Included:

  • Battle Royale: 100-player elimination (too AAA-focused for most readers)
  • MOBA: Team-based competitive (too AAA-focused)
  • CCG Constructed: Card collection games (similar to deck-builder archetype)
  • MMO: Massive multiplayer worlds (too large-scale)
  • Fighting Game: 1v1 combat (very specialized)
  • Rhythm Game: Music-based timing (very specialized)
  • Sports Sim: Realistic sports (very specialized)

What to Do:

  • Find the closest match (might be 50-70% aligned)
  • Build custom using framework (start at Phase 0)
  • Reference multiple archetypes (hybrid approach)
  • Focus on aesthetic profile first (this determines structure)

Conclusion

These archetypes represent validated patterns—accumulated design knowledge from dozens of successful games. Use them as starting points, not constraints. Modify freely to serve your aesthetic goals. When in doubt, return to your aesthetic profile and ask: "Does this archetype serve the experience I want to create?"

The framework (Phases 0-5) works with or without archetypes. Archetypes simply accelerate the process by providing proven defaults. Innovation often comes from thoughtful modification of existing patterns, not pure invention.
